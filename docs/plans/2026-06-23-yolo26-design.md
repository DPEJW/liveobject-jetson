# YOLO26 detection upgrade ‚Äî liveobject-jetson

- **Date:** 2026-06-23
- **Status:** Approved, in implementation

## Goal

Improve detection accuracy on the Orin NX `liveobject` app by replacing YOLO11m
with YOLO26, and publish the Orin/TensorRT version as a new repo
`DPEJW/liveobject-jetson`. The original `DPEJW/liveobject` stays as the
Raspberry Pi / Hailo version.

## Decisions

- Build **both** `yolo26m` and `yolo26l` engines; switch live via the existing UI
  model-switcher.
- Export with **`end2end=False`** (traditional one-to-many head ‚Üí `(1, 84, 8400)`)
  so the engines are drop-in for the existing `trt_yolo.py` decode. The default
  NMS-free head (`(1, 300, 6)`) is avoided: riskier to build on TensorRT 8.5 and
  no accuracy benefit (same mAP, only saves CPU-side NMS).
- Keep `yolo12m` / `yolo11m` listed for A/B comparison.

## Why YOLO26

COCO mAP50-95: `yolo26m` 53.1, `yolo26l` 55.0 (vs `yolo11m` ‚âà51.5). More
parameter-efficient than YOLO11; adds STAL (small-object label assignment); DFL
removed ‚Üí simpler export and better old-TensorRT / edge compatibility.

## Pipeline

1. **Mac (throwaway venv):** `pip install -U ultralytics onnx onnxslim`;
   `yolo export model=yolo26{m,l}.pt format=onnx end2end=False imgsz=640
   opset=17 simplify=True half=False`. Verify ONNX output is `(1, 84, 8400)`.
2. `scp` the two `.onnx` to the Orin.
3. **Orin (ssh):** `trtexec --onnx=yolo26X.onnx --fp16
   --saveEngine=models/yolo26X.engine`. Record inference ms. Fall back
   opset 16 ‚Üí 13 if the TRT 8.5 parser rejects an op.
4. `config.py`: add `yolo26m` / `yolo26l`, default to `yolo26m`. `trt_yolo.py`
   decode reused unchanged (verified against the exported shape, not assumed).
5. Restart `liveobject.service`; verify in browser; A/B all models.

## Risks

- **TensorRT 8.5 build** (main risk) ‚Äî mitigated by opset fallback; verified
  empirically on the Orin, not assumed.
- **fps** ‚Äî `yolo26l` is heavier; pair with MAXN (`nvpmodel -m 0`, needs reboot)
  if sluggish. `yolo26m` ‚âà today's ~8.6 fps @ 15W.
- **Mono camera** ‚Äî grayscale input caps color-dependent accuracy regardless of
  model; the m/l A/B reveals whether the model is actually the bottleneck.
- **One camera = one app** ‚Äî `liveobject.service` `Conflicts=baffle-qc.service`.
